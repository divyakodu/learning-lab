Learn BDD by testing resume-app, three ways

Not a fourth automation library — a different way of writing down what's under test, on top of tools already covered. The same five scenarios, expressed once in Gherkin, wired up three different ways: pytest-bdd, Behave, and Cucumber.js.

~15 minutes · needs resume-app running, Python 3, Node.js, and a real Chrome install

01 What BDD actually buys you

concept

Gherkin is a plain-language format (Given/When/Then) for describing behavior in a .feature file. The point isn't the syntax — it's that the same file a non-programmer can read is the literal thing the test runner executes, not documentation that quietly drifts out of sync with what's actually tested. Each Gherkin line is matched, by its exact wording, to a "step definition" — a real function in Python or JS that does the actual work.

All three tools here execute the identical scenario file:

Feature: Resume list and detail

  Scenario: Downloading a PDF
    When I download the PDF for "divya-kodukula"
    Then the downloaded file should be a valid PDF

Five scenarios total: browsing the list, viewing a resume, the not-found case, and both downloads — same coverage as the other UI stacks, described differently.

02 The same step, wired three ways

Same Gherkin line: Then I should see a link for "divya-kodukula". Three different step-definition conventions underneath.

bdd-pytest-bdd/test_resume.py

scenarios("features/resume.feature")

@then(parsers.parse('I should see a link for "{slug}"'))
def see_link_for(driver, slug):
    link = _wait_visible(driver, f'[data-testid="resume-link-{slug}"]')
    assert link.is_displayed()

A pytest plugin: scenarios(...) auto-generates a real pytest test per scenario. State (the driver) is an ordinary pytest fixture, same pattern as selenium-python. parsers.parse(...) captures the quoted value as a plain function argument.

bdd-behave/features/steps/resume_steps.py

@then('I should see a link for "{slug}"')
def see_link_for(context, slug):
    link = _wait_visible(context.driver, f'[data-testid="resume-link-{slug}"]')
    assert link.is_displayed()

Its own standalone runner (behave, not pytest). No fixture injection — shared state lives on one context object, set up in features/environment.py's before_scenario/after_scenario hooks instead.

bdd-cucumber-js/features/step_definitions/resume.steps.js

Then("I should see a link for {string}", async function (slug) {
  const visible = await this.page.getByTestId(`resume-link-${slug}`).isVisible();
  assert.ok(visible);
});

The JS-ecosystem BDD tool, paired here with Playwright rather than Selenium — so downloads use waitForEvent("download") directly, no CDP workaround needed. State lives on this inside each step, set up in Before/After hooks.

03 What happens when the wording drifts

The single most instructive BDD-specific failure: change a Gherkin step's wording without updating its step definition. All three tools catch it — none of them recover the same way.

Verified across all three, by actually changing "I should see a link for" to "I should see a resume link for" in each stack's feature file and rerunning:

ToolWhat it calls thisWhat it does about it
pytest-bddStepDefinitionNotFoundErrorFails immediately, names the exact missing step text and line number. No suggestion offered.
Behave"undefined"Runs everything else, marks that one scenario undefined, and prints a ready-to-paste @then stub in Behave's decorator style.
Cucumber.js"undefined"Same idea as Behave — runs everything else, then prints a stub using {string} expression syntax and a return 'pending' placeholder.

04 Syntax rules that actually bite beginners

Two more mistakes, verified live and worth seeing before you hit them for real — because they don't fail the same way, and one of them is genuinely misleading.

verified — case sensitivity, and it's sneaky

Gherkin keywords (Given, When, Then, And, But) are case-sensitive, but not in the way you'd expect. Lowercasing one — when I open the resume list instead of When I open the resume list — does not raise a parse error. The file still "collects" fine. But the step silently fails to match against its @when-registered function, so the browser never actually navigates anywhere — and the failure you see is a timeout on the next step (waiting 5 seconds for an element that was never going to appear, since the page never loaded), not a clear error pointing at the real cause. Verified: capitalized, the test passes in 0.7s; lowercased, it fails after a 6-second timeout, every time.

verified — quotes are structural, not decorative

Drop the quotes — Then I should see a link for divya-kodukula instead of for "divya-kodukula" — and you get the opposite experience: an immediate, clear StepDefinitionNotFoundError naming the exact unmatched text. The quotes aren't stylistic; they're what tells the parser "this part is a parameter," matching the "{slug}" placeholder in the step definition. No quotes, no parameter, no match — but at least it tells you so right away.

The pattern worth internalizing: some Gherkin mistakes fail loud and immediately (missing quotes, misspelled step text — see §03 above); others fail quiet and late (wrong keyword case). When a BDD test times out for no obvious reason, checking the exact keyword casing on the step above the one that failed is worth doing before assuming the application is broken.

06 Try it yourself

  • pytest-bdd In features/resume.feature, reword one step without touching test_resume.py. Rerun — that one scenario fails with StepDefinitionNotFoundError, the rest still pass.
  • Behave Same edit in its own features/resume.feature. Rerun — 4 pass, 1 reports "undefined" with a generated step stub.
  • Cucumber.js Same edit again. Rerun — 4 scenarios pass, 1 undefined, with Cucumber's own stub format.
  • pytest-bdd In bdd-pytest-bdd/features/resume.feature, lowercase just the When on the first scenario's first line to when. Rerun — it still collects fine, but that scenario fails after a ~6s timeout instead of pytest-bdd's usual instant, clear error.
  • pytest-bdd In the same file, remove the quotes around "divya-kodukula" in a Then step. Rerun — immediate StepDefinitionNotFoundError, naming the exact unquoted text.

Undo each change and confirm you're back to 5/5 before moving on. Full exercise detail is in each stack's own README.
