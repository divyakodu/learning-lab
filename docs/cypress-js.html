Learn Cypress by testing resume-app's UI

A third answer to "does it work for a user?" — after Playwright (drives the browser from outside) and Selenium (drives it via the WebDriver protocol), Cypress takes a genuinely different architectural approach: it runs inside the browser, alongside your application.

~15 minutes · needs resume-app running and Node.js

0

Before you start

cd resume-app
scripts/start.sh
mkdir cypress-tutorial && cd cypress-tutorial
npm init -y
npm install cypress
concept

Playwright and Selenium both drive a browser from the outside — a separate process sends commands over a protocol (CDP, WebDriver). Cypress instead injects its test runner directly into the browser tab your app is running in. That's why its commands can be so aggressively auto-retrying: there's no network round-trip between "check if this element exists" and the actual DOM.

1

The smallest possible test

cypress.config.js 
const { defineConfig } = require("cypress");
module.exports = defineConfig({
  e2e: { baseUrl: "http://localhost:8080", supportFile: false },
});
cypress/e2e/smoke.cy.js 
describe("resume-app", () => {
  it("loads the homepage", () => {
    cy.visit("/index.html");
    cy.get('[data-testid="resume-list"]').should("be.visible");
  });
});
npx cypress run
Running:  smoke.cy.js

  resume-app
    ✓ loads the homepage

  1 passing (Xms)

The first run downloads Cypress's actual browser-automation binary — around 550MB unpacked, cached globally in ~/Library/Caches/Cypress and shared across every project using the same version, so it's a one-time cost, not a per-project one.

2

Commands retry themselves — you don't ask them to

concept

Neither cy.get(...) nor .should(...) fail instantly if the element isn't there yet — Cypress retries the whole chain for up to 4 seconds by default before giving up. Compare this to the pytest tutorial's explicit assertions or Selenium's explicit WebDriverWait: Cypress's retrying is a global default you opt out of, not a wait you opt into per line.

it("shows the known resume", () => {
  cy.visit("/index.html");
  cy.get('[data-testid="resume-list"]').should("be.visible");
  cy.get('[data-testid="resume-link-divya-kodukula"]').should("be.visible");
});
1 passing

Click, navigate, assert

it("navigates to the detail page on click", () => {
  cy.visit("/index.html");
  cy.get('[data-testid="resume-link-divya-kodukula"]').click();
  cy.url().should("include", "slug=divya-kodukula");
  cy.get('[data-testid="resume-name"]').should("have.text", "Divya Kodukula");
});
1 passing
4

Downloads: the pleasant surprise

concept

Recall the Selenium tutorial: headless Chrome blocks downloads by default, and unblocking them needed real investigation (a Chrome prefs dict, a CDP command). Cypress's bundled Electron browser has no such restriction to begin with — cy.readFile() against the configured downloads folder just works.

it("PDF download produces a real PDF", () => {
  cy.visit("/resume.html?slug=divya-kodukula");
  cy.get('[data-testid="download-pdf"]').click();
  const dir = Cypress.config("downloadsFolder");
  cy.readFile(`${dir}/divya-kodukula.pdf`, "binary", { timeout: 10000 }).should(
    (content) => expect(content.slice(0, 4)).to.eq("%PDF")
  );
});
1 passing

Verified: this passed on the very first run, no workaround needed — a genuine, concrete difference between Cypress's in-browser architecture and Selenium/Playwright's external-driver approach, not just a config preference.
