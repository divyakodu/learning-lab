Learn Playwright by testing resume-app's UI

No prior Playwright knowledge assumed. This is a different kind of testing than the API tutorial — instead of checking what the server returns, you'll drive a real browser and check what a person would actually see and do.

~15 minutes · needs resume-app running and Node.js

0

Before you start

Start resume-app in another terminal, and leave it running for the whole tutorial:

cd resume-app
scripts/start.sh

Then set up a project and install Playwright — it needs an actual browser binary, not just a JS library, so the install step downloads one:

mkdir pw-tutorial && cd pw-tutorial
npm init -y
npm install --save-dev @playwright/test
npx playwright install chromium
1

What Playwright actually does

concept

pytest talks to resume-app's API directly over HTTP. Playwright instead launches a real, invisible (headless) Chromium browser, points it at a URL, and lets you script the same actions a person would take — click, type, wait for things to appear — then makes assertions about what actually rendered. It's testing the same app from the other side: through the screen, not through the wire.

Create tests/smoke.spec.js:

tests/smoke.spec.js 
const { test, expect } = require('@playwright/test');

test('resume-app homepage loads', async ({ page }) => {
  await page.goto('http://localhost:8080/index.html');
  await expect(page).toHaveTitle('Resume App');
});
npx playwright test
Running 1 test using 1 worker

  ✓  1 tests/smoke.spec.js:3:1 › resume-app homepage loads (90ms)

  1 passed (1.6s)

Find things the way a test should

concept

You could find elements by CSS class or button text, but both change for reasons that have nothing to do with the feature breaking — a redesign, a copy edit. resume-app's frontend carries data-testid attributes specifically so tests can reference elements by a stable, intentional name instead. page.getByTestId(...) is how Playwright looks those up.

test('resume list shows the known resume', async ({ page }) => {
  await page.goto('http://localhost:8080/index.html');
  await expect(page.getByTestId('resume-link-divya-kodukula')).toBeVisible();
});
npx playwright test
  ✓  1 tests/smoke.spec.js:3:1 › resume list shows the known resume (183ms)

  1 passed (597ms)

Typo the testid (resume-link-nope) and you'd see: Error: expect(locator).toBeVisible() failed ... Timeout: 5000ms ... element(s) not found. Worth causing once on purpose — Playwright waits the full timeout before giving up, since the element might just be loading slowly, so a "not found" failure takes a few seconds to report, not milliseconds.

3

Script a real interaction

Click the link, and check the browser actually navigated and rendered the right content:

test('clicking a resume link goes to its detail page', async ({ page }) => {
  await page.goto('http://localhost:8080/index.html');
  await page.getByTestId('resume-link-divya-kodukula').click();
  await expect(page).toHaveURL(/resume\.html\?slug=divya-kodukula/);
  await expect(page.getByTestId('resume-name')).toHaveText('Divya Kodukula');
});
npx playwright test
  ✓  1 tests/smoke.spec.js:3:1 › clicking a resume link goes to its detail page (150ms)

  1 passed (429ms)
4

Verify an actual file download

concept

A file download is a browser event, not a page navigation — you have to start listening for it before you click, or you might miss it. That's why the click and the wait are wrapped in Promise.all: both start at the same instant, instead of racing each other.

const fs = require('fs');

test('PDF download button produces a real PDF', async ({ page }) => {
  await page.goto('http://localhost:8080/resume.html?slug=divya-kodukula');
  const [download] = await Promise.all([
    page.waitForEvent('download'),
    page.getByTestId('download-pdf').click(),
  ]);
  const buf = fs.readFileSync(await download.path());
  expect(buf.subarray(0, 4).toString()).toBe('%PDF');
});
npx playwright test
  ✓  1 tests/smoke.spec.js:4:1 › PDF download button produces a real PDF (247ms)

  1 passed (517ms)

Same principle as the API tutorial's PDF test: reading the actual bytes, not just "the click didn't throw," is what catches a download that silently produced an empty or wrong file.

Now break it, on purpose

  • In resume-app/frontend/html-js/js/app.js, change resume-link-${r.slug} to link-${r.slug} in loadList() — no rebuild needed, just refresh. The list test times out after 5s, unable to find the renamed testid.
  • In the same file's loadDetail(), delete the line assigning pdf-link's href. The download test fails — clicking has nowhere to point, so no download event ever fires.

Undo each change and confirm you're back to 7/7 before moving on.
