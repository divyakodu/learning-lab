Learn Locust by load-testing resume-app

A third, genuinely different kind of testing. The pytest and Playwright tutorials both ask "is this correct?" This one asks "does it hold up when many things happen at once?" — a question correctness tests structurally can't answer, no matter how many of them you write.

~15 minutes · needs resume-app running (both replicas) and Python 3

0

Before you start

cd resume-app
scripts/start.sh
mkdir locust-tutorial && cd locust-tutorial
python3 -m venv .venv
.venv/bin/pip install locust
1

The smallest possible load test

concept

A Locust file defines a User class — a template for one simulated visitor. Each @task is something that user does. Locust spawns however many copies of it you ask for and runs them concurrently, then reports on what happened across all of them.

locustfile.py 
from locust import HttpUser, task

class ResumeAppUser(HttpUser):
    @task
    def health(self):
        self.client.get("/api/health")
.venv/bin/locust -f locustfile.py --headless --users 1 --spawn-rate 1 --run-time 5s --host http://localhost:8080
Type     Name                #reqs   #fails |  Avg  Min  Max  Med | req/s  fail/s
GET      /api/health          1668  0(0.00%) |    2    1   15    2 | 477.16    0.00

One simulated user just fired 1,668 requests in 5 seconds. That's not realistic — a real person clicks, reads, then clicks again. Nothing in the code above told Locust to pause, so it hammered the endpoint as fast as the network would allow.

2

Make it behave like real users

Add wait_time — a pause, randomized between two bounds, that Locust inserts after every task:

locustfile.py 
from locust import HttpUser, between, task

class ResumeAppUser(HttpUser):
    wait_time = between(0.5, 2)

    @task
    def health(self):
        self.client.get("/api/health")
.venv/bin/locust -f locustfile.py --headless --users 5 --spawn-rate 5 --run-time 8s --host http://localhost:8080
Type     Name                #reqs   #fails |  Avg  Min  Max  Med | req/s  fail/s
GET      /api/health            32  0(0.00%) |    5    3   10    5 |   4.09    0.00

32 requests from 5 users over 8 seconds — a far more realistic rate. This is the number to tune deliberately, not accept by accident: it's the difference between "simulating traffic" and "simulating a denial-of-service attack against your own app."

3

Simulate a realistic mix of behavior

concept

@task(10) vs @task(2) sets relative frequency — this user is roughly 5x more likely to browse the list than download a PDF, matching how people actually use the app.

locustfile.py 
from locust import HttpUser, between, task

SLUG = "divya-kodukula"

class ResumeAppUser(HttpUser):
    wait_time = between(0.5, 2)

    @task(10)
    def view_list(self):
        self.client.get("/api/resumes", name="/api/resumes")

    @task(2)
    def download_pdf(self):
        self.client.get(f"/api/resumes/{SLUG}/pdf", name="/api/resumes/[slug]/pdf")
.venv/bin/locust -f locustfile.py --headless --users 10 --spawn-rate 5 --run-time 10s --host http://localhost:8080
Type     Name                              #reqs   #fails |  Avg  Min  Max  Med | req/s
GET      /api/resumes                         66  0(0.00%) |    5    1   11    5 |  6.7
GET      /api/resumes/[slug]/pdf               9  0(0.00%) |  143   94  234  120 |  0.9

Roughly the 5:1 ratio the weights implied, and the PDF endpoint is consistently ~25x slower than the list — expected, since it does real rendering work (weasyprint) instead of serving JSON.

why percentiles, not just the average

"Avg" hides outliers — one request that took 3 seconds and nine that took 5ms still average out to a number that looks fine. A percentile answers a sharper question: p90 is the response time that 90% of requests were at or under — so "p90 = 130ms" means one in ten requests was slower than that. That's the number that catches problems an average would hide, as the next step shows.

5

The finding this whole tutorial builds to

Stop one of the two backend replicas — the same infrastructure the load balancer is meant to protect against:

cd resume-app
docker compose stop backend2

Now rerun the load test:

cd learning-lab/load-locust
./run.sh

Verified result: the failure rate stayed at 0% in every run — nginx silently retries any request that hits the dead backend2 onto backend1. But a real, multi-second tail appeared: across several runs, somewhere in the top 1-10% of requests (the exact percentile shifted run to run) spiked to 2-3+ seconds, against a healthy median in the single-digit milliseconds. One run's p90 alone hit 7.6 seconds.

Every request still technically "worked." A functional test suite — api-pytest, or the tutorial before this one — would show all green in this exact state. Only measuring response times under concurrent load reveals that the app is, in practice, in real trouble. That gap is the entire reason load testing exists as its own discipline rather than "more functional tests, run faster."

Restore it and confirm recovery:

cd resume-app
docker compose start backend2
$ ./run.sh
# p90 back down to double-digit milliseconds, 0 failures
6

Keep experimenting

A few directions worth trying on your own, now that the mechanics make sense: raise --users until you find the point where failures start appearing, not just latency; watch what happens if you stop both replicas; or add a @task for /api/resumes/{slug}/regenerate and see how a POST under load compares to a GET.
