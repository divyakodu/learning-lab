Learn Selenium by testing resume-app's UI

Same question as the Playwright tutorial — does it work for a user? — answered with a different, older, more manual tool. The contrast is the point: Selenium won't wait for you the way Playwright does, and that turns out to matter a lot.

~15 minutes · needs resume-app running, Python 3, and a real Chrome install

0

Before you start

cd resume-app
scripts/start.sh
mkdir selenium-tutorial && cd selenium-tutorial
python3 -m venv .venv
.venv/bin/pip install selenium pytest requests
concept

Selenium drives a real browser through the WebDriver protocol — an actual W3C standard, supported by every major browser vendor, not just Chrome. That's Selenium's whole reason for existing: the same test code can, in principle, run against Chrome, Firefox, or Safari by swapping which driver you launch. Modern Selenium (4.6+) includes Selenium Manager, which auto-downloads the matching driver binary for whatever browser you have installed — you still need the actual browser, just not the driver.

The smallest possible test

from selenium import webdriver
from selenium.webdriver.chrome.options import Options

opts = Options()
opts.add_argument("--headless=new")
driver = webdriver.Chrome(options=opts)
driver.get("http://localhost:8080/index.html")
print(driver.title)
driver.quit()
.venv/bin/python3 smoke.py
Resume App

Selenium Manager quietly resolved and launched a matching chromedriver the first time this ran — no separate download step, no version to pin by hand.

Why "just find the element" isn't enough

concept

Playwright's getByTestId(...).click() automatically waits for the element to exist and be actionable. Selenium's find_element(...) does not — it looks right now, and throws NoSuchElementException if the page hasn't rendered the element yet. On a fast local app like resume-app this rarely bites, but on anything with real network latency, a naive test becomes flaky exactly when it matters most.

The fix is to wait explicitly, for a specific condition, with a timeout:

from selenium.webdriver.common.by import By
from selenium.webdriver.support import expected_conditions as EC
from selenium.webdriver.support.ui import WebDriverWait

def test_resume_list_shows_known_resume(driver):
    driver.get("http://localhost:8080/index.html")
    link = WebDriverWait(driver, 5).until(
        EC.visibility_of_element_located(
            (By.CSS_SELECTOR, '[data-testid="resume-link-divya-kodukula"]')
        )
    )
    assert link.is_displayed()
.venv/bin/pytest -v smoke_test.py
smoke_test.py::test_resume_list_shows_known_resume PASSED

1 passed in 0.42s
3

Script a click and check navigation

def test_clicking_resume_link_navigates_to_detail(driver):
    driver.get("http://localhost:8080/index.html")
    link = WebDriverWait(driver, 5).until(
        EC.element_to_be_clickable(
            (By.CSS_SELECTOR, '[data-testid="resume-link-divya-kodukula"]')
        )
    )
    link.click()
    WebDriverWait(driver, 5).until(EC.url_contains("slug=divya-kodukula"))
    name = driver.find_element(By.CSS_SELECTOR, '[data-testid="resume-name"]')
    assert name.text == "Divya Kodukula"
PASSED

Note the plain find_element for reading name.text at the end, with no wait — by that point EC.url_contains already proved the navigation completed, so the element is guaranteed to exist. Waits belong at the moment of uncertainty, not sprinkled everywhere out of habit.

Downloads: the part Selenium makes you do by hand

concept

Playwright exposes page.waitForEvent("download"). Selenium has no equivalent — headless Chrome doesn't even allow downloads by default. Verified two ways to unblock it: a Chrome prefs dict setting a download directory, or the Page.setDownloadBehavior CDP command. On the Chrome version used here, either alone was sufficient — but since headless download behavior has changed across Chrome versions before, using both together is deliberate redundancy, not decoration.

opts.add_experimental_option("prefs", {
    "download.default_directory": download_dir,
    "download.prompt_for_download": False,
})
driver.execute_cdp_cmd(
    "Page.setDownloadBehavior", {"behavior": "allow", "downloadPath": download_dir}
)
button.click()
# ...then poll the filesystem, since there's no download event to await
while not os.listdir(download_dir):
    time.sleep(0.2)

Verified: remove both mechanisms and the click still fires, but no file ever appears within a 10s poll — Chrome has nowhere it's allowed to save it. Remove only one and leave the other, and it still works — that asymmetry is worth seeing once, in step 6.
