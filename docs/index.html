Learning to test resume-app, three ways

learning-lab isn't just three folders of test code — it's the same app, tested with three genuinely different questions in mind. This page walks through what each question is, how the code answers it, and one real finding each approach caught that the others couldn't. New here? Start with resume-app's guide first, so the app itself makes sense before its tests do.

api-pytest 10/10 ui-playwright 7/7 load-locust 0% failures @ 20 users

01 Three questions, three tools

Every test in this repo answers one of three questions. Confusing them is the single most common mistake in test automation — a suite that only ever asks the first question will ship a slow, broken-feeling app with a fully green pipeline.

QuestionToolWhat it actually checks
Does it work?api-pytestThe API returns correct data, correct status codes, correct file bytes — one request at a time, in isolation
Does it work the way a user experiences it?ui-playwrightClicking things in a real browser produces the right visible result, using the same hooks (data-testid) a user's actions would trigger
Does it still work well under load?load-locustResponse times and failure rates when many requests arrive at once — not correctness, capacity

02 One real test from each, annotated

Not simplified examples — the actual test files, and why each assertion is there.

api-pytest/tests/test_downloads.py

def test_pdf_download(base_url, known_slug):
    resp = requests.get(f"{base_url}/api/resumes/{known_slug}/pdf")
    assert resp.status_code == 200
    assert resp.headers["content-type"] == "application/pdf"
    assert resp.content[:4] == b"%PDF"

Why three separate assertions, not one: a status-code-only check would pass even if weasyprint silently failed and the endpoint returned an empty 200. Checking the content-type and the file's actual magic bytes (%PDF) means the test fails loudly the moment PDF generation is broken — which, per the fix log, it once was.

ui-playwright/tests/downloads.spec.ts

test("PDF download button produces a real PDF", async ({ page }) => {
  await page.goto(`/resume.html?slug=${KNOWN_SLUG}`);
  const [download] = await Promise.all([
    page.waitForEvent("download"),
    page.getByTestId("download-pdf").click(),
  ]);
  const buf = fs.readFileSync(await download.path());
  expect(buf.subarray(0, 4).toString()).toBe("%PDF");
});

Why getByTestId, not a CSS selector or button text: a copy change ("Download PDF" → "Get PDF") or a class-name refactor wouldn't break this test — only removing the data-testid attribute itself would, which is a deliberate, visible decision, not an accident. Why read the file's bytes instead of just checking the click didn't throw: a silently-broken download (0-byte file, wrong content) would still "succeed" by that lower bar.

load-locust/locustfile.py

class ResumeAppUser(HttpUser):
    wait_time = between(0.5, 2)

    @task(10)
    def view_list(self):
        self.client.get("/api/resumes", name="/api/resumes")

    @task(2)
    def download_pdf(self):
        self.client.get(f"/api/resumes/{SLUG}/pdf", name="/api/resumes/[slug]/pdf")

Why relative weights (@task(10) vs @task(2)): real traffic isn't uniform — browsing the list is far more common than downloading a PDF, so the simulated load should reflect that mix, not hammer every endpoint equally. See §04 below for what this actually caught.

03 What load testing catches that the others can't

This is the finding worth internalizing more than any single assertion above.

Stopping one of resume-app's two backend replicas (docker compose stop backend2) and re-running api-pytest would show... all 10 tests still passing. nginx silently retries any request that hits the dead replica onto the surviving one, so every response still eventually arrives correct.

Verified finding: running load-locust in that same state — one replica down — showed a 0% failure rate, but p90 latency jumped from ~50ms to ~7.6 seconds, and p99 to 13-14 seconds. Every request "worked." The app was, in practice, unusable.

That gap — 100% functionally correct, and simultaneously in real trouble — is exactly why load testing exists as its own discipline rather than "more of the same tests, but faster." Full numbers and the exact repro steps are in load-locust's README.

04 Which one do I reach for?

You're worried about...Reach for
A field is missing, a status code is wrong, a file's content is corruptapi-pytest
Something a user would click doesn't do what it shouldui-playwright
It works fine alone but might fall over with real traffic, or when part of the infra is degradedload-locust

05 Try it yourself

Every one of these was actually reproduced — broken, observed, reverted — before being written down. Click to see what's verified to happen.

Comment it out in resume-app/backend/python-fastapi/requirements.txt, then docker compose build --no-cache backend1 backend2 && docker compose up -d. Result: test_pdf_download fails — the endpoint 500s. This is the real bug from resume-app's fix log; the pin is the fix.
In resume-app/nginx/nginx.conf's location /api/ block, comment out add_header X-Served-By ...;, then docker compose restart nginx. Result: test_served_by_header_present fails immediately — the header's just gone.
In the same nginx.conf, comment out server backend2:8000;, restart nginx. Result: test_health_round_robins_across_both_backends fails — every request now lands on backend1 alone.
In resume-app/frontend/html-js/js/app.js, change resume-link-${r.slug} to link-${r.slug} in loadList() — no rebuild needed, refresh is enough. Result: the list-rendering test times out after 5s, unable to find getByTestId("resume-link-divya-kodukula").
In the same file's loadDetail(), delete the document.getElementById("pdf-link").href = ... line. Result: the download test fails — clicking has nowhere to point, so no download event ever fires.
cd resume-app && docker compose stop backend2, then rerun load-locust's ./run.sh. Result: failure rate stays 0%, but p90 latency goes from ~50ms to ~7.6s and p99 to 13-14s. See §03 above — this is the headline finding of this whole repo.

06 Setup, quick reference

All three stacks need resume-app running first, and follow the same shape.

cd resume-app && scripts/start.sh   # http://localhost:8080

cd learning-lab/api-pytest      && ./setup.sh && ./run.sh
cd learning-lab/ui-playwright   && ./setup.sh && ./run.sh
cd learning-lab/load-locust     && ./setup.sh && ./run.sh

Each stack's own README has full setup/run detail, override env vars, and last-verified-run output. This page is the map; those READMEs are the territory.
