Three questions, five tutorials

Confusing which question a test answers is the most common mistake in test automation. Three of the five tutorials deliberately answer the same question with different tools — that's not redundant, it's the fastest way to see what each tool's design actually costs and buys you.

QuestionTutorial
Does it work? — correct data, correct status codes, correct file bytes, one request at a timeapi-pytest
Does it work the way a user experiences it? — real clicks in a real browserui-playwright — auto-waiting, built-in download events
selenium-python — explicit waits, manual download handling
cypress-js — runs in-browser, auto-retrying, downloads work out of the box
Does it still work well under load? — throughput and latency with many requests at onceload-locust

Why this distinction is not academic: stopping one of resume-app's two backend replicas leaves every functional test passing — nginx quietly retries onto the survivor. Only the load tutorial's percentile numbers reveal that p90 latency has jumped from milliseconds to seconds. Full repro in the load-locust tutorial, step 5.

Setup, quick reference

All five need resume-app running first.

cd resume-app && scripts/start.sh   # http://localhost:8080

cd learning-lab/api-pytest      && ./setup.sh && ./run.sh
cd learning-lab/ui-playwright   && ./setup.sh && ./run.sh
cd learning-lab/selenium-python && ./setup.sh && ./run.sh
cd learning-lab/cypress-js      && ./setup.sh && ./run.sh
cd learning-lab/load-locust     && ./setup.sh && ./run.sh

Each stack's own README has full setup/run detail and its own "try breaking this on purpose" exercises for after the tutorial.
