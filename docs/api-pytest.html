learning-lab / api-pytest tutorial

tutorial · build along

Learn pytest by testing resume-app's API

No prior pytest knowledge assumed. You'll write a handful of tests from scratch against a real running app, one concept at a time, ending at the actual suite that lives in this repo.

~15 minutes · needs resume-app running and Python 3

0

Before you start

Start resume-app in another terminal, and leave it running for the whole tutorial:

cd resume-app
scripts/start.sh

Then set up an isolated place to work — a virtual environment (venv) so the packages you install here don't affect anything else on your machine:

mkdir pytest-tutorial && cd pytest-tutorial
python3 -m venv .venv
.venv/bin/pip install pytest requests
1

What pytest actually does

concept

pytest looks for files named test_*.py, runs every function inside them named test_*, and treats each one as pass/fail based on whether its assert statements are true. That's the entire mental model — no special syntax, no test-class boilerplate required.

Create test_smoke.py:

test_smoke.py 
def test_true():
    assert True

Run it:

.venv/bin/pytest -v
test_smoke.py::test_true PASSED                                          [100%]

1 passed in 0.00s
2

Watch one fail, on purpose

This is worth doing before writing anything real — you want to recognize this output the first time it happens for real, not the first time it's confusing. Change the file to:

test_smoke.py 
def test_math():
    assert 2 + 2 == 5
.venv/bin/pytest -v
test_smoke.py::test_math FAILED                                          [100%]

=================================== FAILURES ===================================
__________________________________ test_math ___________________________________

    def test_math():
>       assert 2 + 2 == 5
E       assert (2 + 2) == 5

test_smoke.py:2: AssertionError
1 failed in 0.01s

pytest shows you the exact expression that failed, with the real computed value substituted in ((2 + 2) == 5) — you rarely need a debugger to figure out why a test failed; the failure output usually already tells you.

3

Test something real

Replace the file with a test against resume-app's actual health endpoint, using the requests library to make a real HTTP call:

test_smoke.py 
import requests

def test_health():
    resp = requests.get("http://localhost:8080/api/health")
    assert resp.status_code == 200
.venv/bin/pytest -v
test_smoke.py::test_health PASSED                                        [100%]

1 passed in 0.32s

If resume-app isn't running, you'd see requests.exceptions.ConnectionError: ... Connection refused instead — worth seeing once now so it's recognizable later. It means the test's code was never the problem; there was nothing listening on that port.

4

Stop repeating the URL

concept

A fixture is reusable setup that pytest hands to any test function that asks for it by parameter name — pytest sees a parameter called base_url, finds a fixture with that name, calls it, and passes the result in. It lives in a special file, conftest.py, that pytest loads automatically.

conftest.py 
import pytest

@pytest.fixture
def base_url():
    return "http://localhost:8080"
test_smoke.py 
import requests

def test_health(base_url):
    resp = requests.get(f"{base_url}/api/health")
    assert resp.status_code == 200
    assert resp.json()["status"] == "ok"
.venv/bin/pytest -v
test_smoke.py::test_health PASSED                                        [100%]

1 passed in 0.05s

Now every test in the file can just ask for base_url instead of hardcoding it — and if you ever needed to point at a different environment, you'd change it in exactly one place.

5

Build a real test, one assertion at a time

Create test_pdf.py and start with the simplest possible check:

test_pdf.py 
import requests

def test_pdf_download(base_url):
    resp = requests.get(f"{base_url}/api/resumes/divya-kodukula/pdf")
    assert resp.status_code == 200
.venv/bin/pytest test_pdf.py -v
test_pdf.py::test_pdf_download PASSED                                    [100%]

1 passed in 0.20s

That passes — but it would also pass if the server crashed and returned a 200 with garbage in the body, or the wrong file type entirely. Layer on two more checks:

test_pdf.py 
import requests

def test_pdf_download(base_url):
    resp = requests.get(f"{base_url}/api/resumes/divya-kodukula/pdf")
    assert resp.status_code == 200
    assert resp.headers["content-type"] == "application/pdf"
    assert resp.content[:4] == b"%PDF"
.venv/bin/pytest test_pdf.py -v
test_pdf.py::test_pdf_download PASSED                                    [100%]

1 passed in 0.19s
why three checks, not one

Each assertion guards a different failure mode: status_code catches a crash or error response; content-type catches the server returning the right status but the wrong kind of response (say, an HTML error page with a 200); the magic-bytes check (b"%PDF" is how every real PDF file starts) catches a response that looks fine on the surface but is actually empty or corrupted. None of the three implies the others.

Test the unhappy path too

A suite that only checks things working isn't finished — add a case for a resume that doesn't exist:

test_pdf.py 
def test_pdf_unknown_slug_is_404(base_url):
    resp = requests.get(f"{base_url}/api/resumes/does-not-exist/pdf")
    assert resp.status_code == 404
.venv/bin/pytest test_pdf.py -v
test_pdf.py::test_pdf_download PASSED                                    [ 50%]
test_pdf.py::test_pdf_unknown_slug_is_404 PASSED                         [100%]

2 passed in 0.22s

Now break it, on purpose

Reinforce what each assertion is actually for by making it fail, in the real suite this time:

  • Comment out pydyf==0.10.0 in resume-app/backend/python-fastapi/requirements.txt, rebuild (docker compose build --no-cache backend1 backend2 && docker compose up -d). test_pdf_download fails — a real bug this pin fixes.
  • Comment out add_header X-Served-By ...; in resume-app/nginx/nginx.conf, docker compose restart nginx. test_served_by_header_present fails.
  • Comment out server backend2:8000; in the same file's upstream block, restart nginx. test_health_round_robins_across_both_backends fails — only backend1 ever answers.

Undo each change and confirm you're back to 10/10 before moving on.
